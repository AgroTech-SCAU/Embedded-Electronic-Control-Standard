# 嵌入式电控通用开发流标准

> 适用对象：AgroTech 协会电控组成员，开发基于 MCU 的裸机/RTOS 工程下的底盘电控、机械臂电控、执行机构控制、传感器接入、通信协议与运动控制项目

> 核心目标：让不同项目在目录结构、接口边界、开发顺序、调试方法、复用 SDK 方式上保持一致，减少“能跑但不可维护”的代码

---

## 0. 完整开发流程

一个新电控模块或新项目从 0 到 1 时，推荐按以下流程推进；每一步都对应本文后续章节，不建议跳过接口、安全和测试直接进入 `while(1)` 调试

1. [确认基本理念](#1-本标准的基本理念)：先分层、先接口、先安全，再写具体实现
2. [建立统一分层结构](#2-统一分层结构)：按 `app/service/device/domain/infra/platform` 放置代码
3. [划清各层职责边界](#3-各层职责边界)：确认功能属于 app、service、device、domain、infra 还是 platform
4. [按从 0 到 1 的开发流程建模块](#4-从-0-到-1-的开发流程)：定义需求、安全边界、public header、底层连接方式和测试入口
5. [按单模块设计标准写接口](#5-单模块设计标准)：先确定错误码、配置、状态、生命周期、PortOps 和调用方式
6. [按模块间解耦和层级协作组织依赖](#6-模块间解耦和层级协作)：让 platform 能力通过 service 或 adapter 进入 device/infra
7. [组织上层任务流](#7-上层任务流组织方法)：从简单循环升级为 app task、command dispatcher、service update 和 HFSM
8. [按一般开发顺序落地代码](#8-一般开发顺序)：从需求、安全、platform、infra、device、domain、service 到 app 逐步闭环
9. [按模块完成标准验收](#9-模块完成标准-definition-of-done)：检查接口、错误码、安全、测试、示例和跨层依赖
10. [按 SDK 分类标准沉淀复用模块](#10-sdk-分类标准)：将通用能力放入 `sdks/infra`、`sdks/domain`、`sdks/device`
11. [同步 README 和项目说明](#11-readme-编写标准)：让使用者知道如何编译、接线、初始化和测试
12. [按迁移建议整理旧项目](#12-对现有代码的迁移建议)：小步迁移，不一次性推翻可运行版本
13. [执行安全约束](#13-安全约束)：电机、舵机、底盘、机械臂必须先有 stop、timeout、fault 和限幅
14. [按 PR 验收清单合并](#14-示例-pr-验收清单)：提交前说明影响层级、测试记录、安全影响和文档同步

可参考的贯穿式示例：

- [`examples/module_design/portops_motor_device/`](examples/module_design/portops_motor_device/)：从需求边界、public header、PortOps、mock port 到 service 绑定的最小电机设备示例

## 1. 本标准的基本理念

嵌入式电控代码不应只追求“这次比赛能跑”，而应沉淀为可教学、可迁移、可复用、可调试的工程资产

**本标准优先保证**：

1. 先分层，再写代码：业务流程、系统服务、真实设备、数学模型、基础设施、芯片适配必须分开
2. 先接口，再实现：先定义 `.h`、错误码、数据结构、生命周期和调用边界，再写 `.c`
3. 先安全，再功能：涉及电机、舵机、底盘、机械臂时，停止、限幅、超时、失联、急停必须先设计
4. 硬件相关与硬件无关隔离：能脱离真实硬件的模块，尽量能在 PC/mock 环境测试
5. 解耦服务于复用和测试：PortOps、callback、接口表、X-Macro、HFSM 都是为降低维护成本服务
6. 重构必须可验证：每次迁移、拆层、抽象都要保留最小测试记录

## 2. 统一分层结构

**AgroTech 协会嵌入式电控工程统一采用以下从上到下的分层结构**：

```text
src/
├── app/        # 1. 应用层：任务入口、业务流程、状态机编排、整机逻辑
├── service/    # 2. 服务层：组合 device + domain + infra，形成系统能力
├── device/     # 3. 设备层：真实设备 SDK、设备协议、反馈缓存、设备抽象
├── domain/     # 4. 领域层：运动学、控制模型、机构解算、纯算法模型
├── infra/      # 5. 基础设施层：PID、滤波、矩阵、协议解析、HFSM、CRC、容器
└── platform/   # 6. 平台层：芯片/HAL/FSP/CubeMX 外设适配、CAN/UART/PWM/GPIO/Tick
```

**基本依赖规则**：

- 下层不得依赖上层
- app 只依赖 service 和少量 infra
- service 可依赖 device/domain/infra，但不应把平台句柄暴露给 app
- device 可依赖 domain/infra，对 platform 应优先通过 PortOps 或注册函数连接
- domain 可依赖 infra，但不应依赖 device/platform
- infra 应尽量无业务和芯片依赖，必要时通过 PortOps 或注册函数连接 platform
- platform 是唯一允许直接包含 HAL/FSP/CMSIS/CubeMX 头文件的层

**另外**：
- service 负责 device/infra 与 platform 之间的对接，并完成系统能力的组合、缓存和安全策略
- 如需提升性能或确定设备无法复用，则允许 device 直接依赖 platform，但应限制在内部实现，不暴露给上层

## 3. 各层职责边界

### 3.1 app/：应用层

> 职责：描述整机要做什么

**典型内容**：

- `entry_init()` / `entry_loop()`
- 主任务状态机
- 业务模式切换
- 上电初始化顺序编排
- 人机交互入口
- command dispatcher
- 整机故障处理入口
- 比赛/项目业务逻辑

**允许依赖**：

- `service/`
- 少量 `infra/`，例如 HFSM、logger、非阻塞 delay

**禁止事项**：

- 不直接操作 HAL/FSP/CubeMX 句柄
- 不直接拼 CAN/UART/SPI 帧
- 不直接写复杂运动学、轨迹规划或控制算法
- 不把所有逻辑塞进 `main.c` 或 `while(1)`
- 不维护设备协议细节

### 3.2 service/：服务层

> 职责：描述系统能力如何组合

service 是 app 与底层模块之间的系统能力层；它负责把算法、设备、反馈缓存、安全策略、控制周期组合成一个可被 app 调用的能力；同时还负责在自身 `init` 中完成 device/infra 与 platform 的对接

**典型内容**：

- `chassis_controller`：将底盘速度指令转换为各轮速度/转角，并调用电机与舵机
- `arm_controller`：组合 IK/FK、轨迹、关节电机、限位和安全停止
- `pick_service`：组合视觉目标、机械臂运动、夹爪动作
- `communication_service`：组合协议解析、链路状态、业务消息分发
- `safety_service`：组合急停、失联、超时、限幅与降级输出
- device/infra PortOps 绑定：把 platform 提供的 CAN/UART/Tick/GPIO 等函数填入 device/infra 的 init 配置

**允许依赖**：

- `device/`
- `domain/`
- `infra/`
- platform 的窄接口或 adapter

**禁止事项**：

- 不直接把平台句柄暴露给 app
- 不让 app 知道设备协议
- 不写成纯转发层，service 应承担组合、缓存、安全、调度职责

**示例接口**：

```c
/* chassis_service.h */
typedef enum {
    CHASSIS_OK = 0,
    CHASSIS_INVALID_PARAM,
    CHASSIS_INVALID_MODEL,
    CHASSIS_DEVICE_ERROR,
    CHASSIS_TIMEOUT,
} ChassisStatus;

ChassisStatus chassis_init(float length, float width, float wheel_radius, float max_speed);
ChassisStatus chassis_set_velocity(float vx, float vy, float wz);
ChassisStatus chassis_stop(void);
ChassisStatus chassis_update(void);
```

`chassis_init()` 不只初始化 service 自己的状态；它还应统一完成 motor device、parser/logger 等 infra 模块与 platform 的 PortOps 对接，避免 app 或 main 分散调用底层绑定函数

### 3.3 device/：设备层

> 职责：描述某个真实设备如何被控制

device 关注真实硬件设备，但不应把自己写死在某一个 MCU 的 HAL 上；正式 SDK 应通过 PortOps 与 `platform/` 对接；如果没有特殊原因，PortOps 应作为 init 配置的一部分传入，避免 `register_port()` 和 `init()` 分开调用导致漏绑定

**典型内容**：

- 电机 SDK
- 舵机 SDK
- 编码器 SDK
- IMU SDK
- 通信模块 SDK
- 设备反馈缓存
- 设备状态解析
- 设备超时与掉线判断

**允许依赖**：

- `domain/`
- `infra/`
- platform 提供的 PortOps 或注册函数

**禁止事项**：

- 不写业务判断，例如“采摘完成”“巡线开始”
- 不在中断里执行复杂控制逻辑
- 不将 HAL 全局句柄到处散落
- 不把芯片头文件扩散到通用 SDK 的 public header

**示例 PortOps**：

```c
/* motor_device.h */
#include <stdbool.h>

typedef struct {
    bool (*send)(uint32_t id, const uint8_t* data, uint8_t len);
    uint32_t (*now_ms)(void);
    void (*register_rx)(void (*cb)(uint32_t id, const uint8_t* data, uint8_t len));
} MotorPortOps;

typedef struct {
    const MotorPortOps* ops;
    uint32_t feedback_timeout_ms;
} MotorConfig;

MotorStatus motor_init(const MotorConfig* config);
MotorStatus motor_set_speed(uint8_t id, float speed_rad_s);
MotorStatus motor_get_feedback(uint8_t id, MotorFeedback* out);
```

### 3.4 domain/：领域层

> 职责：描述数学模型和领域算法

domain 应尽量与真实硬件无关，最好能在 PC 上编译和测试

**典型内容**：

- 舵轮/轮轨/阿克曼/麦轮运动学
- 机械臂 FK/IK
- 坐标变换
- 轨迹规划
- 底盘模型
- 控制量限幅
- 夹持力模型
- 机构几何解算

**允许依赖**：

- `infra/`
- 可在目标平台使用的标准 C/C++ 库

**禁止事项**：

- 不包含 `main.h`、`stm32xxx_hal.h`、`hal_data.h` 等芯片头文件
- 不直接调用 CAN/UART/GPIO
- 不直接读写真实设备
- 不处理业务状态机
- 不把单位写模糊，必须明确 m/mm、rad/deg、m/s、rad/s、rpm 等单位

### 3.5 infra/：基础设施层

> 职责：提供与业务和具体硬件无关的通用能力

**典型内容**：

- PID、滤波、限幅器
- ring buffer、CRC、协议解析器
- HFSM
- 矩阵运算
- 通用错误码
- logger 抽象
- 时间差工具
- 单元测试工具

**要求**：

- 尽量无 HAL 依赖
- 尽量无全局硬件句柄
- 尽量能在 PC 上测试
- 数据结构自包含
- 错误码清晰
- 如果需要 Tick、临界区、输出流等底层能力，应像 device 一样通过 PortOps 接入 platform，并尽量集成到 init 配置中

**示例 infra PortOps**：

```c
/* timeout_guard.h */
typedef struct {
    uint32_t (*now_ms)(void);
} TimeoutGuardPortOps;

typedef struct {
    const TimeoutGuardPortOps* ops;
    uint32_t timeout_ms;
} TimeoutGuardConfig;

TimeoutGuardStatus timeout_guard_init(TimeoutGuard* self, const TimeoutGuardConfig* config);
TimeoutGuardStatus timeout_guard_refresh(TimeoutGuard* self);
bool timeout_guard_is_timeout(const TimeoutGuard* self);
```

### 3.6 platform/：平台层

> 职责：封装芯片和板级外设能力

**典型内容**：

- CAN/FDCAN、UART/USART、SPI/I2C
- GPIO、TIMER/PWM、DMA、SysTick
- 临界区、cache、NVIC、中断
- retarget/printf
- 板级初始化

**要求**：

- 允许包含 HAL/FSP/CMSIS 头文件
- 对上提供稳定、窄接口
- 只做底层能力封装，不写业务逻辑
- 中断回调只搬运数据、置标志、分发 callback，不执行复杂控制
- 提供回调函数注册入口，例如 `can_rx_callback_register()`、`uart_rx_callback_register()`

## 4. 从 0 到 1 的开发流程

**一个新模块或新功能从 0 到 1 时，不应从 `while(1)` 或 HAL 调用开始，而应按以下顺序推进**：

1. 定义需求和安全边界：控制对象、输入输出、最大输出、默认状态、失联处理、急停链路
2. 判断所属层级：app/service/device/domain/infra/platform
3. 定义 public header：错误码、配置、状态、反馈、命令、生命周期函数
4. 定义实现边界：哪些内容隐藏在 `.c` 或 private header，哪些允许上层依赖
5. 定义底层连接方式：纯函数、handle/context、接口单例、PortOps、callback 或 message/event
6. 定义测试入口：PC mock、单模块测试、板级测试、真实硬件低速测试
7. 先跑最小闭环，再接入完整 app 任务流

**判断模块层级时优先使用以下问题**：

| 问题 | 倾向层级 |
|---|---|
| 是否描述整机任务、模式切换、故障恢复 | app |
| 是否组合设备、算法、反馈和安全策略 | service |
| 是否控制真实设备或解析真实设备反馈 | device |
| 是否是运动学、坐标系、轨迹、限幅等硬件无关算法 | domain |
| 是否是 PID、parser、HFSM、CRC、容器等通用工具 | infra |
| 是否直接接触 HAL/FSP/CubeMX、芯片句柄、中断或 DMA | platform |

## 5. 单模块设计标准

### 5.1 模块设计检查表

拿到一个模块需求后，第一步是定义边界，而不是写实现

| 设计项 | 必须回答的问题 |
|---|---|
| 模块职责 | 这个模块只解决什么问题，不解决什么问题 |
| 所属层级 | 放在 app/service/device/domain/infra/platform 哪一层 |
| 输入输出 | 输入来自哪里，输出交给谁，单位和坐标系是什么 |
| 数据结构 | 配置、状态、反馈、命令、错误码如何组织 |
| 生命周期 | 是否需要 init/start/update/stop/reset/deinit |
| 调用方式 | 同步调用、周期 update、callback、中断触发还是事件驱动 |
| 错误模型 | 参数错误、状态错误、超时、底层失败、不可恢复故障如何表达 |
| 安全策略 | 限幅、默认输出、超时停机、失联保护、故障锁存如何处理 |
| 可测试性 | 能否 PC mock，如何做单元测试、板级测试、真实硬件测试 |
| 接口边界 | public header 暴露什么，private 实现隐藏什么 |

### 5.2 最小 public header 骨架

```c
/* module.h */
typedef enum {
    MODULE_OK = 0,
    MODULE_INVALID_PARAM,
    MODULE_INVALID_STATE,
    MODULE_TIMEOUT,
    MODULE_PORT_ERROR,
} ModuleStatus;

typedef struct {
    float max_output;
    uint32_t timeout_ms;
} ModuleConfig;

typedef struct Module Module;

ModuleStatus module_init(Module* self, const ModuleConfig* config);
ModuleStatus module_update(Module* self);
ModuleStatus module_stop(Module* self);
ModuleStatus module_get_status(const Module* self, ModuleStatus* out);
```

public header 只暴露上层需要依赖的类型和函数；内部缓存、协议细节、平台句柄、私有状态应留在 `.c` 或 private header 中

### 5.3 设计方法选择表

| 方法 | 适合场景 | 不适合场景 | 示例 |
|---|---|---|---|
| 纯函数模块 | 无状态算法、单位转换、几何计算 | 需要保存状态或生命周期 | angle normalize、unit conversion |
| 多实例对象 + handle/context | 同类对象有多个实例，每个实例有独立状态 | 系统只有唯一入口且不需要并行实例 | PID、filter、parser、kinematics model |
| 接口单例 | 系统需要唯一入口，上层希望统一调用 real/mock/sim | 同时存在多个独立实例 | motor manager、bus_servo manager、logger |
| PortOps 注册函数 | SDK 要跨平台复用，底层由 platform 或 chip SDK 提供 | 模块完全不接底层 | motor、IMU、UART device |
| callback 注册 | 底层异步上报、通信接收、中断分发 | 需要严格同步返回 | CAN RX、UART RX、按键事件 |
| X-Macro 表驱动 | 错误码、命令表、状态表、设备表统一维护 | 表很小且不会扩展，或逻辑本身很复杂 | error code、command dispatcher |
| HFSM | 上层任务流程复杂，有嵌套状态、公共父状态和异常跳转 | 简单单步控制 | app task、pick flow、chassis mode |
| service 组合器 | 需要组合设备、算法、反馈、安全逻辑 | 单纯设备协议封装 | chassis_controller、arm_controller |

### 5.4 PortOps 与注册函数

注册函数的作用是把“谁来提供底层能力”从模块实现中抽出来；PortOps 是一组函数指针表，用于描述模块需要 platform 提供哪些能力

在本标准中，device/infra 不应直接依赖 `hfdcan1`、`huart1`、`HAL_GetTick()` 等平台对象；它们只声明自己需要的能力，例如 `send`、`now_ms`、`lock`、`unlock`、`write`

**优先推荐**：

```text
module_init(config with PortOps)
```

**不优先推荐**：

```text
module_register_port(ops)
module_init(config)
```

**原因**：

- 分成 register 和 init 后容易漏调用
- init 时无法一次性检查依赖是否完整
- 测试代码和真实项目容易出现两套初始化顺序
- service 无法清晰表达“这个系统能力已经完成底层绑定”

只有在确实需要运行期热切换底层实现、或模块生命周期长于某个 port 生命周期时，才保留单独 `set_port()` / `register_port()` 接口

### 5.5 init 接口设计原则

初版设计允许只设计必要的 `.h` 骨架；在 `.c` 实现过程中，如果发现缺少配置项、状态查询、错误码或测试入口，可以按需扩展 `.h`

**扩展 `.h` 时必须满足**：

- 新增字段有明确职责
- 新增接口有明确调用时机
- 不把平台句柄暴露给 public header
- 不为了临时调试暴露内部缓存
- 改动 public API 时同步更新文档和示例

**推荐 init 形态**：

```c
/* module.h */
typedef struct {
    const ModulePortOps* ops;
    uint32_t timeout_ms;
    float max_output;
} ModuleConfig;

ModuleStatus module_init(Module* self, const ModuleConfig* config);
```

**init 至少负责**：

- 参数检查
- PortOps 完整性检查
- 配置保存
- 内部状态清零
- 默认安全输出或默认安全状态设置
- 必要 callback 注册

### 5.6 device PortOps 示例实现

```c
/* motor_device.h */
typedef enum {
    MOTOR_OK = 0,
    MOTOR_INVALID_PARAM,
    MOTOR_PORT_ERROR,
    MOTOR_TIMEOUT,
} MotorStatus;

typedef struct {
    bool (*send)(uint32_t id, const uint8_t* data, uint8_t len);
    uint32_t (*now_ms)(void);
    void (*register_rx)(void (*cb)(uint32_t id, const uint8_t* data, uint8_t len));
} MotorPortOps;

typedef struct {
    const MotorPortOps* ops;
    uint32_t feedback_timeout_ms;
} MotorConfig;

MotorStatus motor_init(const MotorConfig* config);
MotorStatus motor_set_speed(uint8_t id, float speed_rad_s);
MotorStatus motor_stop(uint8_t id);
```

/* motor_device.c */
static const MotorPortOps* s_ops;
static uint32_t s_feedback_timeout_ms;

    (void)id;
    (void)data;

}

### 5.7 infra PortOps 示例实现

```c
/* timeout_guard.h */
typedef enum {
    TIMEOUT_GUARD_OK = 0,
    TIMEOUT_GUARD_INVALID_PARAM,
    TIMEOUT_GUARD_PORT_ERROR,
} TimeoutGuardStatus;

typedef struct {
    uint32_t (*now_ms)(void);
} TimeoutGuardPortOps;

typedef struct {
    const TimeoutGuardPortOps* ops;
    uint32_t timeout_ms;
} TimeoutGuardConfig;

typedef struct {
    const TimeoutGuardPortOps* ops;
    uint32_t timeout_ms;
    uint32_t last_refresh_ms;
} TimeoutGuard;

TimeoutGuardStatus timeout_guard_init(TimeoutGuard* self, const TimeoutGuardConfig* config);
TimeoutGuardStatus timeout_guard_refresh(TimeoutGuard* self);
bool timeout_guard_is_timeout(const TimeoutGuard* self);
```

```c
/* timeout_guard.c */
TimeoutGuardStatus timeout_guard_init(TimeoutGuard* self, const TimeoutGuardConfig* config) {
    if(self == NULL || config == NULL || config->ops == NULL) {
        return TIMEOUT_GUARD_INVALID_PARAM;
    }

    if(config->ops->now_ms == NULL) {
        return TIMEOUT_GUARD_PORT_ERROR;
    }

    self->ops = config->ops;
    self->timeout_ms = config->timeout_ms;
    self->last_refresh_ms = config->ops->now_ms();

    return TIMEOUT_GUARD_OK;
}

TimeoutGuardStatus timeout_guard_refresh(TimeoutGuard* self) {
    if(self == NULL || self->ops == NULL || self->ops->now_ms == NULL) {
        return TIMEOUT_GUARD_INVALID_PARAM;
    }

    self->last_refresh_ms = self->ops->now_ms();
    return TIMEOUT_GUARD_OK;
}

bool timeout_guard_is_timeout(const TimeoutGuard* self) {
    uint32_t now;

    if(self == NULL || self->ops == NULL || self->ops->now_ms == NULL) {
        return true;
    }

    now = self->ops->now_ms();
    return (now - self->last_refresh_ms) > self->timeout_ms;
}
```

### 5.8 C 风格 OOP 的边界

C 语言中不要为了“像类”而强行复制完整 OOP；应优先选择能降低复杂度的对象形态

| 对象形态 | 使用边界 |
|---|---|
| 入口单例 + 数据单例 | 系统天然只有一个对象，例如全局 logger；风险是隐藏状态过多 |
| 数据多例 | 算法或工具对象天然有多个实例，例如 PID/filter/parser |
| 入口单例 + 数据多例 | 上层需要统一入口，但内部管理多个设备，例如 motor manager |
| 入口多例 + 数据多例 | 只有在多个实现确实需要并存时使用，避免过度抽象 |

**多实例对象最小骨架**：

```c
/* pid.h */
typedef struct {
    float kp;
    float ki;
    float kd;
    float integral;
} Pid;

void pid_init(Pid* self, float kp, float ki, float kd);
float pid_update(Pid* self, float target, float feedback);
```

**接口单例最小骨架**：

```c
/* module_interface.h */
typedef struct {
    ModuleStatus (*init)(void);
    ModuleStatus (*stop)(void);
    ModuleStatus (*update)(void);
} ModuleInterface;

extern const ModuleInterface module_real;
extern const ModuleInterface module_mock;
extern const ModuleInterface* module_instance;

void module_set_instance(const ModuleInterface* instance);

#define module (*module_instance)
```

**常见越界**：

- 为了像成员函数，给每个对象都挂一张方法表
- 为了模拟 private，滥用 `void* private`，导致类型边界更差
- 把所有模块都抽成同一种接口风格
- 抽象层只改变写法，没有减少依赖、测试或替换成本

### 5.9 X-Macro 的使用边界

X-Macro 适合维护“天然就是表”的内容，例如错误码、命令表、状态表、设备表；它的价值是让枚举、字符串、分发表从同一份表生成，避免漏改

```c
/* command_table.h */
#define COMMAND_TABLE \
    X(CMD_STOP, handle_stop, "stop all actuators") \
    X(CMD_SET_VEL, handle_set_vel, "set chassis velocity") \
    X(CMD_CLEAR_FAULT, handle_clear_fault, "clear latched fault")

#define X(name, handler, desc) name,
typedef enum {
    COMMAND_TABLE
} CommandId;
#undef X

#define X(name, handler, desc) static CommandStatus handler(const CommandFrame* frame);
COMMAND_TABLE
#undef X
```

不应把复杂业务逻辑塞进 X-Macro；表负责描述关系，逻辑仍应放在普通函数中

## 6. 模块间解耦和层级协作

### 6.1 解耦方法

解耦不是为了炫技，而是为了换平台、换设备、可测试和可维护

| 方法 | 解耦目标 |
|---|---|
| public/private header | 上层只依赖稳定接口，不依赖内部实现 |
| PortOps | device/infra 与 platform 分离，便于跨芯片复用 |
| interface table | real/mock/sim 实现切换，便于测试 |
| callback | 底层异步事件向上层上报，中断只做轻量分发 |
| message/event | app 不直接调用深层细节，任务流通过事件驱动 |
| service | app 不直接操作 device/domain，由 service 在 init 中完成 PortOps 绑定并组合系统能力 |
| 配置结构体 | 参数集中管理，避免散落在代码中 |
| 错误码 | 下层失败不直接把 HAL/FSP 状态传给上层 |

**反例**：

- app 直接调用 `HAL_CAN_AddTxMessage()`
- service 中到处引用 `hfdcan1`
- device SDK 的 public header 包含 `stm32xxx_hal.h`
- 中断回调里执行复杂控制逻辑
- 一个全局变量同时被 app/device/platform 修改

### 6.2 接口流、数据流、错误流

| 层级 | 暴露接口 | 隐藏实现 | 可依赖 | 不应依赖 | 测试方式 |
|---|---|---|---|---|---|
| platform | 窄 Port 函数、底层 callback 注册 | HAL/FSP 句柄、中断、DMA | 芯片 SDK、板级代码 | app/service/device/domain | 板级外设测试 |
| infra | 通用工具 API | 内部状态和算法细节 | 标准 C、可选 platform PortOps | 业务和芯片头文件 | PC 单测 |
| domain | 数学模型和解算 API | 具体算法缓存 | infra | device/platform | PC 单测、边界测试 |
| device | 设备命令、反馈、超时、stop/fault | 协议帧和反馈缓存 | domain/infra、PortOps | app、平台句柄暴露 | mock port、单设备低速 |
| service | 系统能力 API、周期 update | 设备绑定、控制周期、安全策略 | device/domain/infra | app 业务细节、平台句柄暴露 | 闭环测试 |
| app | 任务入口、命令分发、模式状态机 | 业务流程内部状态 | service、少量 infra | HAL、设备协议 | 整机流程和异常测试 |

**初始化推荐顺序**：

```text
platform bringup
→ domain model init
→ service init
  → service 组装 device/infra PortOps
  → service 调用 device/infra init
  → service 初始化自身状态和安全策略
→ app task start
```

**错误流推荐规则**：

```text
platform error
→ device/infra 转换成本层错误码
→ service 聚合为系统状态和安全动作
→ app 做模式切换、故障锁存、恢复入口
```

### 6.3 service init 统一绑定示例

service 的 `init` 是系统能力的装配入口；app 不需要知道 motor 使用 CAN，timeout guard 使用 Tick，也不需要知道这些能力来自哪个 HAL 句柄

```c
/* chassis_service.h */
typedef enum {
    CHASSIS_OK = 0,
    CHASSIS_PORT_ERROR,
    CHASSIS_DEVICE_ERROR,
    CHASSIS_INFRA_ERROR,
} ChassisStatus;

ChassisStatus chassis_init(void);
ChassisStatus chassis_set_velocity(float vx, float vy, float wz);
ChassisStatus chassis_update(void);
ChassisStatus chassis_stop(void);
```

```c
/* chassis_service.c */
#include "chassis_service.h"
#include "motor_device.h"
#include "timeout_guard.h"
#include "platform_can.h"
#include "platform_time.h"

static TimeoutGuard s_cmd_timeout;

static const MotorPortOps s_motor_ops = {
    .send = platform_can_send,
    .now_ms = platform_time_now_ms,
    .register_rx = platform_can_register_rx,
};

static const TimeoutGuardPortOps s_timeout_ops = {
    .now_ms = platform_time_now_ms,
};

ChassisStatus chassis_init(void) {
    MotorConfig motor_config = {
        .ops = &s_motor_ops,
        .feedback_timeout_ms = 50u,
    };

    TimeoutGuardConfig timeout_config = {
        .ops = &s_timeout_ops,
        .timeout_ms = 200u,
    };

    if(motor_init(&motor_config) != MOTOR_OK) {
        return CHASSIS_DEVICE_ERROR;
    }

    if(timeout_guard_init(&s_cmd_timeout, &timeout_config) != TIMEOUT_GUARD_OK) {
        return CHASSIS_INFRA_ERROR;
    }

    return CHASSIS_OK;
}
```

**这个写法的关键点**：

- platform 函数只在 service 的 `.c` 中出现
- app 只调用 `chassis_init()`，不会漏掉 motor 或 timeout guard 的底层绑定
- device/infra 仍然不知道 HAL 句柄，也不知道项目使用的是 STM32、Renesas 还是 PC mock

### 6.4 adapter 对接方式

如果 service 没法直接把 platform 函数传给 device/infra，需要增加 adapter；adapter 的职责是把“不匹配的平台接口”转换成 device/infra 需要的 PortOps 形状

**适合使用 adapter 的场景**：

- platform 函数签名和 device/infra PortOps 不一致
- 需要在发送前补默认 CAN 通道、总线号、互斥锁或统计信息
- chip SDK 已经提供一套端口接口，但名称或参数不符合当前通用 SDK
- 需要把多个 platform 能力组合成一个更窄的 PortOps

```c
/* chassis_motor_adapter.h */
#include "motor_device.h"

const MotorPortOps* chassis_motor_adapter_get_ops(void);
```

```c
/* chassis_motor_adapter.c */
#include "chassis_motor_adapter.h"
#include <stdbool.h>
#include "platform_usart.h"
#include "platform_time.h"

#define MAX_RX_LISTENERS 5

typedef void(*RxCb)(uint32_t id, const uint8_t* data, uint8_t len);

typedef struct {
    RxCb cb;
    bool active;
} listener_t;

static listener_t listeners[MAX_RX_LISTENERS];
static bool wrapper_registered;

static void platform_rx_wrapper(const uint8_t* data, uint8_t len) {
    if(len == 0) return;
    uint32_t id = data[0];
    const uint8_t *real_data = data + 1;
    uint8_t real_len = len - 1;
    for(uint8_t i = 0; i < MAX_RX_LISTENERS; i++) {
        if(listeners[i].active && listeners[i].cb) {
            listeners[i].cb(id, real_data, real_len);
        }
    }
}

static bool adapter_motor_send(uint32_t id, const uint8_t* data, uint8_t len) {
    uint8_t tx_buf[256];
    if(len + 1 > sizeof(tx_buf)) return false;
    tx_buf[0] = (uint8_t)(id & 0xFF);
    for(uint8_t i = 0; i < len; i++) {
        tx_buf[i + 1] = data[i];
    }
    return platform_usart1_send(tx_buf, len + 1);
}

static void adapter_motor_register_rx(void(*cb)(uint32_t id, const uint8_t* data, uint8_t len)) {
    if(!cb) return;
    for(uint8_t i = 0; i < MAX_RX_LISTENERS; i++) {
        if (listeners[i].active && listeners[i].cb == cb) return;
    }
    for(uint8_t i = 0; i < MAX_RX_LISTENERS; i++) {
        if(!listeners[i].active) {
            listeners[i].cb = cb;
            listeners[i].active = true;
            if(!wrapper_registered) {
                platform_usart1_register_rx(platform_rx_wrapper);
                wrapper_registered = true;
            }
            return;
        }
    }
}

static const MotorPortOps s_ops = {
    .send = adapter_motor_send,
    .now_ms = platform_time_now_ms,
    .register_rx = adapter_motor_register_rx,
};

const MotorPortOps* chassis_motor_adapter_get_ops(void) {
    return &s_ops;
}
```

```c
/* chassis_service.c */
#include "chassis_motor_adapter.h"

ChassisStatus chassis_init(void) {
    MotorConfig motor_config = {
        .ops = chassis_motor_adapter_get_ops(),
        .feedback_timeout_ms = 50u,
    };

    if(motor_init(&motor_config) != MOTOR_OK) {
        return CHASSIS_DEVICE_ERROR;
    }

    return CHASSIS_OK;
}
```

adapter 不应变成新的业务层；它只负责接口形状转换，不负责底盘模式、速度解算、故障策略或任务流程

## 7. 上层任务流组织方法

**简单项目常从以下写法开始**：

```c
while (1) {
    read_cmd();
    control_motor();
    check_error();
}
```

**当任务开始包含模式切换、故障恢复、异步命令和安全抢占时，应升级为**：

```text
app task
→ command dispatcher
→ service API
→ service update loop
→ device/domain/infra
→ platform PortOps
```

**基本分工**：

- app task 描述“要做什么”，不描述底层如何发帧
- command dispatcher 负责外部命令到内部动作的映射
- service update 负责周期控制、反馈处理、限幅和安全输出
- HFSM 负责复杂任务流程、异常中断、恢复和共享父状态
- safety service 或 safety guard 负责抢占式 stop/brake/fault
- stop/brake/fault 优先级必须高于普通任务

**HFSM 状态设计应至少定义四类函数**：

```c
/* app_state.h */
typedef struct {
    void (*entry)(void);
    void (*exit)(void);
    void (*action)(void);
    void (*handle_event)(int event);
} AppStateOps;
```

**适合 HFSM 的状态示例**：

```text
底盘模式：IDLE / MANUAL / AUTO / FAULT
机械臂流程：INIT / HOMING / READY / MOVING / HOLDING / FAULT
采摘流程：SEARCH / APPROACH / GRASP / PICK / PLACE / RECOVER
```

如果只有一个简单开关动作，不需要 HFSM；普通函数或简单 FSM 更清楚

## 8. 一般开发顺序

### Step 0：需求与安全边界

**写代码前必须明确**：

- 控制对象是什么
- 输入是什么
- 输出是什么
- 最大速度、最大电流、最大角度、最大加速度
- 失联后如何处理
- 上电默认状态
- 急停逻辑
- 是否允许直接失能
- 是否需要回零、回中、刹车
- 出错后能否自动恢复
- 是否存在夹人、砸落、飞车、过流、烧驱动风险

### Step 1：定义目录与接口

**先建立目录**：

```text
src/app
src/service
src/device
src/domain
src/infra
src/platform
```

**再定义头文件接口**：

- 类型
- 状态码
- 单位
- 函数表
- 注册函数
- 初始化顺序
- 生命周期
- 可重入/不可重入说明
- 周期调用说明

原则：先写 `.h`，再写 `.c`

### Step 2：打通 platform/

**优先验证**：

- 时钟树
- 串口日志
- LED
- 系统 tick
- CAN/UART/PWM
- 中断回调
- 基础发送与接收

**要求**：

- 每个外设必须有最小测试
- 每个通信外设必须有发送、接收、错误回调
- 中断回调必须支持注册分发
- 不要一开始就接复杂设备，先验证总线本身可用

### Step 3：接入 infra/

**优先接入**：

- delay/time
- ring buffer
- parser
- CRC
- logger
- HFSM
- PID/matrix/filter

**要求**：

- 可独立测试
- 与 HAL 无关
- 提供 example
- 给出边界测试
- 能在 PC 上跑的尽量先在 PC 上跑

### Step 4：开发 device/

**开发顺序**：

1. 明确设备协议
2. 定义设备状态码
3. 定义设备反馈结构
4. 定义包含 PortOps 的 init 配置
5. 实现命令编码
6. 实现反馈解析
7. 实现反馈缓存
8. 实现超时
9. 实现初始化、停止、失能
10. 实现最小闭环测试

**设备层必须考虑**：

- ID 合法性
- 参数范围
- 总线发送失败
- 反馈超时
- 重复回调注册
- 主动上报与查询上报并存
- 上电初始状态
- 掉线后的安全输出

### Step 5：开发 domain/

**开发顺序**：

1. 明确数学模型
2. 明确坐标系和单位
3. 明确输入输出
4. 明确边界条件
5. 写 PC 测试
6. 再接入 service

**domain 必须写清楚**：

- 长度单位：m/mm
- 角度单位：rad/deg
- 速度单位：m/s、rad/s、rpm
- 坐标系方向
- 正方向定义
- 解算失败条件
- 限幅策略

### Step 6：开发 service/

**开发顺序**：

1. 引入 platform 的窄接口或 adapter
2. 组装 device/infra 需要的 PortOps
3. 在 service init 中调用 device/infra init
4. 组合 domain 与 device
5. 建立状态缓存
6. 建立控制周期
7. 处理设备反馈
8. 处理超时
9. 实现 stop/brake/fault
10. 暴露给 app 使用

**service 必须考虑**：

- app 调用频率
- service 内部更新频率
- device/infra 的 PortOps 是否已在 service init 中完成绑定
- 是否需要 adapter 转换 platform 接口
- 反馈是否及时
- 输出是否限幅
- 底层设备失败后的返回值
- 安全停止是否可靠

### Step 7：开发 app/

app 层最后写；app 应该调用已经稳定的 service，而不是边写业务边直接改底层

**app 开发顺序**：

1. 上电初始化
2. 模式切换
3. 任务状态机
4. 用户输入/通信命令
5. 故障处理
6. 日志输出
7. 整机联调

### Step 8：联调与测试

**联调顺序**：

1. PC mock 测试
2. 单模块静态测试
3. 单外设收发测试
4. 单设备低速测试
5. service 闭环测试
6. app 整机流程测试
7. 长时间稳定性测试
8. 异常场景测试：断线、掉电、失联、堵转、限位、非法指令

## 10. SDK 分类标准

**AgroTech 协会通用 SDK 统一放在 `Embedded-Electronic-Control-Standard/sdks/` 中**：

```text
sdks/
├── infra/     # 通用基础设施 SDK
├── domain/    # 领域/算法 SDK
└── device/    # 常用设备 SDK
```

### 10.1 infra：通用基础设施 SDK

**特点**：

- 与真实硬件无关
- 不直接依赖 HAL/FSP/CubeMX
- 尽量可在 PC/mock 环境测试
- 可被 device/domain/service 复用

**典型模块**：

- ring buffer
- protocol parser
- CRC
- PID
- filter
- HFSM
- matrix
- logger interface
- delay/time abstraction

### 10.2 domain：领域/算法 SDK

**特点**：

- 与真实硬件无关
- 不需要对接 CAN/UART/GPIO
- 必须明确单位、坐标系、输入输出范围和失败条件
- 尽量可以单独进行 PC 测试

**典型模块**：

- 机械臂 FK/IK
- 舵轮/轮轨/阿克曼运动学
- trajectory profile
- limiter
- angle normalize
- coordinate transform
- unit conversion helper

### 10.3 device：常用设备 SDK

**特点**：

- 与真实设备有关
- 不应写死某一个 MCU
- 通过 PortOps/注册函数与 platform 对接
- public header 不应包含芯片 HAL/FSP 头文件

**典型模块**：

- motor
- bus_servo
- encoder
- IMU
- CAN sensor
- UART module

### 10.4 chip SDK 不放在本仓库内

**STM32、Renesas、ESP32、GD32 等具体芯片平台适配独立为 chip SDK 仓库**：

```text
Embedded-Chip-STM32-HAL-SDK
Embedded-Chip-Renesas-FSP-SDK
```

本标准仓库负责说明“架构怎么写、SDK 怎么设计”；chip SDK 负责说明“某个芯片平台怎么把 CAN/UART/Tick/GPIO/PWM 等底层能力适配成 PortOps”

## 11. README 编写标准

### 11.1 标准仓库 README 应包含

- 标准仓库定位
- 快速导航
- 统一分层结构
- `sdks/` 当前内容

- 测试记录位置

## 12. 对现有代码的迁移建议

**对于已有项目，不建议一次性大改，而是逐步迁移**：

1. 先补 README 和目录说明
2. 把 HAL 直接调用收拢到 platform
3. 把通用工具移入 infra
4. 把数学模型移入 domain
5. 把真实设备协议移入 device
6. 把组合控制逻辑移入 service
7. 最后整理 app 状态机

**迁移时优先保证**：

- 不破坏当前可运行版本
- 每次 PR 只移动或重构一个模块
- 每次重构都有最小验证记录
- 重构前后接口行为一致

---

## 13. 安全约束

**涉及执行机构的模块必须明确**：

- 默认上电输出是什么
- 初始化失败是否输出
- 通信失联如何处理
- 反馈超时如何处理
- 急停是否绕过 app/service
- 是否允许直接失能
- 失能前是否需要回零/回中/刹车
- 速度、电流、角度是否限幅
- 故障恢复是否需要人工确认

**禁止**：

- 没有 stop/brake/fault 逻辑就上真实电机
- 没有反馈超时就闭环
- 没有限幅就接收外部速度/位置命令
- app 直接发送底层控制帧
- 中断中执行复杂控制和阻塞等待
- 机械臂/底盘项目在不确认安全边界时直接高速测试
