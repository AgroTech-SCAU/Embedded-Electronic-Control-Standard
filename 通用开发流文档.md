# 	嵌入式电控通用开发流标准

> **适用对象**：AgroTech 协会电控组成员，开发基于 MCU 的裸机/RTOS 工程下的底盘电控、机械臂电控、执行机构控制、传感器接入、通信协议与运动控制项目

> **核心目标**：让不同项目在目录结构、接口边界、开发顺序、调试方法、复用 SDK 方式上保持一致，减少“能跑但不可维护”的代码

---

## 0. 本标准的基本理念

嵌入式电控代码不应只追求“这次比赛能跑”，而应沉淀为可教学、可迁移、可复用、可调试的工程资产

因此本标准优先保证：

1. **先分层，再写代码**：业务流程、系统服务、真实设备、数学模型、基础设施、芯片适配必须分开
2. **先接口，再实现**：先写 `.h`、错误码、数据结构、生命周期的可维护+扩展骨架，再写 `.c` 的具体实现
3. **先安全，再功能**：涉及电机、舵机、底盘、机械臂时，停止、限幅、超时、失联、急停必须先设计
4. **硬件相关与硬件无关隔离**：能脱离真实硬件的模块，尽量能在 PC/mock 环境测试
5. **模块尽量做到解耦**：不急的情况下尽量将模块解耦的，通过注册函数由上层对接
6. **重构必须可验证**：每次迁移、拆层、抽象都要保留最小测试记录

---

## 1. 统一分层结构

AgroTech 协会嵌入式电控工程统一采用以下从上到下的分层结构：

```text
src/
├── app/        # 1. 应用层：任务入口、业务流程、状态机编排、整机逻辑
├── service/    # 2. 服务层：组合 device + domain，形成系统能力
├── device/     # 3. 设备层：真实设备 SDK、设备协议、反馈缓存、设备抽象
├── domain/     # 4. 领域层：运动学、控制模型、机构解算、纯算法模型
├── infra/      # 5. 基础设施层：PID、滤波、矩阵、协议解析、HFSM、CRC、容器
└── platform/   # 6. 平台层：芯片/HAL/FSP/CubeMX 外设适配、CAN/UART/PWM/GPIO/Tick
```

**基本要求**：

- 下层不得依赖上层；
- 应用层只依赖服务层和少量基础设施层;
- 服务层可完全依赖下层，但不应直接调用平台层；
- 设备层可完全依赖领域层和基础设施层，对平台层应该通过注册函数进行对接；
- 领域层可完全依赖基础设施层，但不应依赖平台层；
- 基础设施层对平台层应该通过注册函数进行对接；
- 服务层负责设备层/基础设施层与平台层的对接；
- 如需提升性能或确定设备无法复用，则允许设备层直接依赖平台层，但应限制在内部实现，不暴露给上层

## 2. 各层职责边界

### 2.1 app/：应用层

> **职责：描述整机要做什么**

**典型内容**：

- `entry_init()` / `entry_loop()`；
- 主任务状态机；
- 业务模式切换；
- 上电初始化顺序编排；
- 人机交互入口；
- 命令分发；
- 整机故障处理入口；
- 比赛/项目业务逻辑

**允许依赖**：

- `service/`；
- 少量 `infra/`，例如 HFSM、logger、非阻塞 delay

禁止事项：

- 不直接操作 HAL/FSP/CubeMX 句柄；
- 不直接拼 CAN/UART/SPI 帧；
- 不直接写复杂运动学、轨迹规划或控制算法；
- 不把所有逻辑塞进 `main.c` 或 `while(1)`；
- 不在 app 中维护设备协议细节

**示例文件**：

```text
app/
├── entry.h / entry.c
├── app_main.c
├── task_fsm.c
├── task_fsm.h
├── command_dispatcher.c
└── command_dispatcher.h
```

---

### 2.2 service/：服务层

> **职责：描述系统能力如何组合**

service 是 app 与底层模块之间的系统胶水层；它负责设备层/基础设施层与平台层的对接，并把算法、设备、反馈缓存、安全策略、控制周期组合成一个可被 app 调用的能力

**典型内容**：

- `chassis_controller`：将底盘速度指令转换为各轮速度/转角，并调用电机与舵机；
- `arm_controller`：组合 IK/FK、轨迹、关节电机、限位和安全停止；
- `pick_service`：组合视觉目标、机械臂运动、夹爪动作；
- `communication_service`：组合协议解析、链路状态、业务消息分发；
- `safety_service`：组合急停、失联、超时、限幅与降级输出

**允许依赖**：

- `domain/`；
- `device/`；
- `infra/`

**禁止事项**：

- 不直接调用平台层；
- 不把具体芯片句柄暴露给上层；
- 不让 app 知道设备协议；
- 不把 service 写成纯转发层，service 应该承担组合、缓存、安全、调度职责

**示例接口**：

```c
typedef enum {
    CHASSIS_OK = 0,
    CHASSIS_INVALID_PARAM,
    CHASSIS_INVALID_MODEL,
    CHASSIS_DEVICE_ERROR,
    CHASSIS_TIMEOUT,
} ChassisStatus;

ChassisStatus chassis_init(float length, float width, float wheel_radius, float max_speed);
ChassisStatus chassis_set_velocity(float vx, float vy, float wz);
ChassisStatus chassis_stop(void);
ChassisStatus chassis_update(void);
```

---

### 2.3 device/：设备层

> **职责：描述某个真实设备如何被控制**

device 关注真实硬件设备，但不应把自己写死在某一个 MCU 的 HAL 上；正式 SDK 应该通过注册函数或 PortOps 与 `platform/` 对接

**典型内容**：

- 电机 SDK；
- 舵机 SDK；
- 编码器 SDK；
- IMU SDK；
- 通信模块 SDK；
- 传感器 SDK；
- 设备反馈缓存；
- 设备状态解析；
- 设备超时与掉线判断

**允许依赖**：

- `domain/`

- `infra/`

**禁止事项**：

- 不在设备层写业务判断，例如“采摘完成”“巡线开始”；
- 不在中断里执行复杂控制逻辑；
- 不将 HAL 全局句柄到处散落；
- 不把芯片头文件扩散到通用 SDK 的 public header

**示例 PortOps 写法**：

```c
typedef struct {
    int (*send)(uint32_t id, const uint8_t* data, uint8_t len);
    uint32_t (*now_ms)(void);
    void (*register_rx)(void (*cb)(uint32_t id, const uint8_t* data, uint8_t len));
} MotorPortOps;

MotorStatus motor_register_port(const MotorPortOps* ops);
MotorStatus motor_init(void);
MotorStatus motor_set_speed(uint8_t id, float speed_rad_s);
MotorStatus motor_get_feedback(uint8_t id, MotorFeedback* out);
```

临时项目允许 device 直接接收 HAL 句柄，但进入AgroTech 协会标准 SDK 前应逐步改成 PortOps

---

### 2.4 domain/：领域层

> **职责：描述数学模型和领域算法**

domain 应尽量与真实硬件无关，最好能在 PC 上编译和测试

**典型内容**：

- 舵轮/轮轨/阿克曼/麦轮运动学；
- 机械臂 FK/IK；
- 坐标变换；
- 轨迹规划；
- 底盘模型；
- 控制量限幅；
- 夹持力模型；
- 机构几何解算

**允许依赖**：

- `infra/`；
- 可在目标平台使用的标准 C/C++ 库

**禁止事项**：

- 不包含 `main.h`、`stm32xxx_hal.h`、`hal_data.h` 等芯片头文件；
- 不直接调用 CAN/UART/GPIO；
- 不直接读写真实设备；
- 不处理业务状态机；
- 不把单位写模糊，必须明确单位

**示例接口**：

```c
typedef struct {
    float length;
    float width;
    float wheel_radius;
    float max_wheel_linear_speed;
} SteerWheelModel;

SteerWheelErrorCode steer_wheel_init(SteerWheel* self, SteerWheelModel model);
SteerWheelErrorCode steer_wheel_ik(SteerWheel* self);
SteerWheelErrorCode steer_wheel_fk(SteerWheel* self);
```

---

### 2.5 infra/：基础设施层

> **职责：提供与业务和具体硬件无关的通用能力**

**典型内容**：

- PID；
- 一阶低通、滑动平均、限幅器；
- ring buffer；
- 帧协议解析器；
- CRC；
- HFSM；
- 矩阵运算；
- 通用错误码；
- logger 抽象；
- 时间差工具；
- 单元测试工具

**要求**：

- 尽量无 HAL 依赖；
- 尽量无全局硬件句柄；
- 尽量能在 PC 上测试；
- 数据结构自包含；
- 错误码清晰

---

### 2.6 platform/：平台层

> **职责：封装芯片和板级外设能力**

典型内容：

- CAN/FDCAN；
- UART/USART；
- SPI/I2C；
- GPIO；
- TIMER/PWM；
- DMA；
- SysTick/时基；
- 临界区；
- cache/NVIC/中断；
- retarget/printf；
- 板级初始化

**要求**：

- 允许包含 HAL/FSP/CMSIS 头文件；
- 对上提供稳定、窄接口；
- 只做底层能力封装，不写业务逻辑；
- 中断回调只搬运数据、置标志、分发 callback，不执行复杂控制；
- 提供回调函数注册入口，例如 `can_rx_callback_register()`、`uart_rx_callback_register()`

## 3. 模块接口设计标准

### 3.1 先写 `.h` 骨架，再写 `.c`，后续可按需补充 `.h`

**每个模块开发顺序**：

1. 定义模块职责；
2. 定义数据结构；
3. 定义错误码；
4. 定义接口函数；
5. 定义初始化顺序；
6. 定义周期调用规则；
7. 定义中断/回调关系；
8. 定义测试方式；
9. 最后写实现
10. `.h` 可按需补充扩展

---

### 3.2 错误码使用 X-Macro 管理

**示例写法**：

```c
#define MOTOR_STATUS_TABLE \
    X(OK, 0, "OK") \
    X(ERROR, 1, "Error") \
    X(TIMEOUT, 2, "Timeout") \
    X(PARAM_INVALID, 3, "Invalid Parameter") \
    X(NO_DATA, 4, "No Data")

#define X(name, value, str) MOTOR_STATUS_##name = value,
typedef enum {
    MOTOR_STATUS_TABLE
} MotorStatus;
#undef X

const char* motor_status_str(MotorStatus status);
```

**优点**：

- 错误码集中维护；
- 枚举、字符串、调试输出可由同一张表生成；
- 减少遗漏和不一致

---

### 3.3 单例接口适用边界

**适合使用接口单例的情况**：

- 系统中逻辑上只有一个入口；
- 上层需要统一调用一组能力；
- 模块需要隐藏内部状态；
- 需要在 real/mock/sim 之间切换实现；
- 典型如 `motor`、`servo`、`chassis`、`hfsm` 的接口表

**不适合使用单例的情况**：

- 同类对象天然有多个实例；
- 每个实例有独立生命周期；
- 需要并行测试多个对象；
- 单例只是为了少传参数；
- 模块内部状态会被多个任务并发访问且没有保护

**示例形式**：

```c
typedef struct {
    MotorStatus (*init)(void);
    MotorStatus (*set_speed)(uint8_t id, float speed_rad_s);
    MotorStatus (*get_feedback)(uint8_t id, MotorFeedback* out);
} MotorInterface;

extern const MotorInterface dm_motor_instance;
extern const MotorInterface* motor_instance;

void motor_set_instance(const MotorInterface* instance);

#define motor (*motor_instance)
```

**使用注意**：

- `set_instance()` 只应在初始化阶段调用；
- 默认实例必须明确；
- mock 实例只能用于测试或仿真；
- 不要让单例接口掩盖真实依赖关系

---

### 3.4 注册函数与 PortOps 适用边界

设备 SDK 如果需要对接底层 CAN/UART/SPI，应该通过注册函数连接平台层：

```c
typedef struct {
    int (*send)(uint32_t id, const uint8_t* data, uint8_t len);
    void (*register_rx)(void (*cb)(uint32_t id, const uint8_t* data, uint8_t len));
    uint32_t (*now_ms)(void);
} DevicePortOps;

DeviceStatus device_register_port(const DevicePortOps* ops);
```

**适合场景**：

- 同一设备 SDK 要在 STM32、Renesas、ESP32 或 PC mock 上复用；
- 底层驱动由芯片工程生成，不希望 SDK 直接依赖 HAL；
- 设备层需要测试发送帧和解析反馈；
- 后续要做仿真或 HIL 测试

---

## 4. 一般开发顺序

### Step 0：需求与安全边界

**写代码前必须明确**：

- 控制对象是什么；
- 输入是什么；
- 输出是什么；
- 最大速度、最大电流、最大角度、最大加速度；
- 失联后如何处理；
- 上电默认状态；
- 急停逻辑；
- 是否允许直接失能；
- 是否需要回零、回中、刹车；
- 出错后能否自动恢复；
- 是否存在夹人、砸落、飞车、过流、烧驱动风险

---

### Step 1：定义目录与接口

**先建立目录**：

```text
src/app
src/service
src/device
src/domain
src/infra
src/platform
```

**再定义头文件接口**：

- 类型；
- 状态码；
- 单位；
- 函数表；
- 注册函数；
- 初始化顺序；
- 生命周期；
- 可重入/不可重入说明；
- 周期调用说明

原则：先写 `.h`，再写 `.c`

---

### Step 2：打通 platform/

**优先验证**：

- 时钟树；
- 串口日志；
- LED；
- 系统 tick；
- CAN/UART/PWM；
- 中断回调；
- 基础发送与接收

**要求**：

- 每个外设必须有最小测试；
- 每个通信外设必须有发送、接收、错误回调；
- 中断回调必须支持注册分发；
- 不要一开始就接复杂设备，先验证总线本身可用

---

### Step 3：接入 infra/

**优先接入**：

- delay/time；
- ring buffer；
- parser；
- CRC；
- logger；
- HFSM；
- PID/matrix/filter

**要求**：

- 可独立测试；
- 与 HAL 无关；
- 提供 example；
- 给出边界测试；
- 能在 PC 上跑的尽量先在 PC 上跑

---

### Step 4：开发 device/

**开发顺序**：

1. 明确设备协议；
2. 定义设备状态码；
3. 定义设备反馈结构；
4. 注册底层 PortOps；
5. 实现命令编码；
6. 实现反馈解析；
7. 实现反馈缓存；
8. 实现超时；
9. 实现初始化、停止、失能；
10. 实现最小闭环测试

**设备层必须考虑**：

- ID 合法性；
- 参数范围；
- 总线发送失败；
- 反馈超时；
- 重复回调注册；
- 主动上报与查询上报并存；
- 上电初始状态；
- 掉线后的安全输出

---

### Step 5：开发 domain/

**开发顺序**：

1. 明确数学模型；
2. 明确坐标系和单位；
3. 明确输入输出；
4. 明确边界条件；
5. 写 PC 测试；
6. 再接入 service

**domain 必须写清楚**：

- 长度单位：m/mm；
- 角度单位：rad/deg；
- 速度单位：m/s、rad/s、rpm；
- 坐标系方向；
- 正方向定义；
- 解算失败条件；
- 限幅策略

---

### Step 6：开发 service/

**开发顺序**：

1. 实现设备层/基础设施层与平台层的对接；

2. 组合 domain 与 device；

3. 建立状态缓存；

4. 建立控制周期；

5. 处理设备反馈；

6. 处理超时；

7. 实现 stop/brake/fault；

8. 暴露给 app 使用

**service 必须考虑**：

- app 调用频率；
- service 内部更新频率；
- 反馈是否及时；
- 输出是否限幅；
- 底层设备失败后的返回值；
- 安全停止是否可靠

---

### Step 7：开发 app/

> app 层最后写，app 应该调用已经稳定的 service，而不是边写业务边直接改底层

**app 开发顺序**：

1. 上电初始化；
2. 模式切换；
3. 任务状态机；
4. 用户输入/通信命令；
5. 故障处理；
6. 日志输出；
7. 整机联调

---

### Step 8：联调与测试

**联调顺序**：

1. PC mock 测试；
2. 单模块静态测试；
3. 单外设收发测试；
4. 单设备低速测试；
5. service 闭环测试；
6. app 整机流程测试；
7. 长时间稳定性测试；
8. 异常场景测试：断线、掉电、失联、堵转、限位、非法指令

---

## 5. 模块完成标准 Definition of Done

一个模块只有满足以下条件，才算完成：

- [ ] 有明确职责说明；
- [ ] 有 `.h` 接口；
- [ ] 有错误码；
- [ ] 有参数检查；
- [ ] 有初始化函数；
- [ ] 有状态查询或反馈接口；
- [ ] 有 stop/deinit/fault 处理，若涉及执行机构；
- [ ] 有最小 example；
- [ ] 有至少一条测试记录；
- [ ] 无明显跨层依赖；
- [ ] 真实硬件相关内容被限制在 device/platform；
- [ ] app 不直接依赖 HAL；
- [ ] 关键安全路径已验证

---

## 6. README.md 编写标准

1. 标准仓库 README：用于说明AgroTech 协会标准是什么、怎么阅读、怎么被项目作为 submodule 引入；
2. 项目仓库 README：用于说明某个具体项目的硬件、软件、编译、烧录、调试和安全注意事项

### 6.1 标准仓库 README 应包含

- 标准仓库定位；
- 快速导航；
- 统一分层结构；
- `sdks/` 当前内容；
- 成员项目如何通过 submodule 引入本标准仓库；
- 普通成员如何更新；
- 项目维护者如何升级标准版本；
- chip SDK 如何按需额外引入；
- 安全提醒；
- 网站同步说明

### 6.2 项目仓库 README 应包含

- 项目目标；
- 硬件组成；
- 软件架构；
- 目录结构；
- 如何初始化 submodule；
- 编译方式；
- 烧录方式；

- 安全注意事项；
- 当前状态；
- 测试记录位置

## 7. 对现有代码的迁移建议

**对于已有项目，不建议一次性大改，而是逐步迁移**：

1. 先补 README 和目录说明；
2. 把 HAL 直接调用收拢到 platform；
3. 把通用工具移入 infra；
4. 把数学模型移入 domain；
5. 把真实设备协议移入 device；
6. 把组合控制逻辑移入 service；
7. 最后整理 app 状态机

**迁移时优先保证**：

- 不破坏当前可运行版本；
- 每次 PR 只移动或重构一个模块；
- 每次重构都有最小验证记录；
- 重构前后接口行为一致

## 8. SDK 分类标准

AgroTech 协会通用 SDK 统一放在 `Embedded-Electronic-Control-Standard/sdks/` 中：

```text
sdks/
├── infra/     # 通用基础设施 SDK
├── domain/    # 领域/算法 SDK
└── device/    # 常用设备 SDK
```

### 8.1 infra：通用基础设施 SDK

**特点**：

- 与真实硬件无关；
- 不直接依赖 HAL/FSP/CubeMX；
- 尽量可在 PC/mock 环境测试；
- 可被 device/domain/service 复用

**典型模块**：

- ring buffer；
- protocol parser；
- CRC；
- PID；
- filter；
- HFSM；
- matrix；
- logger interface；
- delay/time abstraction

### 8.2 domain：领域/算法 SDK

**特点**：

- 与真实硬件无关；
- 不需要对接 CAN/UART/GPIO；
- 必须明确单位、坐标系、输入输出范围和失败条件；
- 尽量可以单独进行 PC 测试

**典型模块**：

- 机械臂 FK/IK；
- 舵轮/轮轨/阿克曼运动学；
- trajectory profile；
- limiter；
- angle normalize；
- coordinate transform；
- unit conversion helper

### 8.3 device：常用设备 SDK

**特点**：

- 与真实设备有关；
- 不应写死某一个 MCU；
- 通过 PortOps/注册函数与 platform 对接；
- public header 不应包含芯片 HAL/FSP 头文件

**典型模块**：

- motor；
- servo；
- encoder；
- IMU；
- CAN sensor；
- UART module

### 8.4 chip SDK 不放在本仓库内

STM32、Renesas、ESP32、GD32 等具体芯片平台适配独立为 chip SDK 仓库：

```text
Embedded-Chip-STM32-HAL-SDK
Embedded-Chip-Renesas-FSP-SDK
```

成员项目根据实际平台额外添加对应 chip SDK submodule

> 本标准仓库负责说明“架构怎么写、SDK 怎么设计”；

> chip SDK 负责说明“某个芯片平台怎么把 CAN/UART/Tick/GPIO/PWM 等底层能力适配成 PortOps”

### 8.5 标准仓库与项目工程的关系

成员项目不建议直接把标准仓库的所有文件复制进项目，而是建议通过 Git submodule 引入：

```text
My-Embedded-Project/
├── external/
│   ├── Embedded-Electronic-Control-Standard/
│   └── Embedded-Chip-STM32-HAL-SDK/
├── src/
│   ├── app/
│   ├── service/
│   ├── device/
│   ├── domain/
│   ├── infra/
│   └── platform/
└── README.md
```

**这样做的好处是**：

1. 项目可以锁定某个标准版本；
2. 标准仓库可以独立演进；
3. 项目更新标准时有明确 PR 记录；
4. 通用 SDK 与具体项目代码边界清楚；
5. chip SDK 可以根据平台按需选择

## 9. 安全约束

**涉及执行机构的模块必须明确**：

- 默认上电输出是什么；
- 初始化失败是否输出；
- 通信失联如何处理；
- 反馈超时如何处理；
- 急停是否绕过 app/service；
- 是否允许直接失能；
- 失能前是否需要回零/回中/刹车；
- 速度、电流、角度是否限幅；
- 故障恢复是否需要人工确认

**禁止**：

- 没有 stop/brake/fault 逻辑就上真实电机；
- 没有反馈超时就闭环；
- 没有限幅就接收外部速度/位置命令；
- app 直接发送底层控制帧；
- 中断中执行复杂控制和阻塞等待；
- 机械臂/底盘项目在不确认安全边界时直接高速测试
