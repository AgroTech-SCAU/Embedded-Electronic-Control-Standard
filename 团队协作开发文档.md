# 嵌入式电控团队协作开发标准

> 适用对象：AgroTech 协会电控组成员、参与AgroTech 协会内项目开发的协作者

> 目标：用统一的 Git/GitHub 工作流管理代码、文档、SDK、Issue、PR、版本和网站同步，避免“谁都能改、谁都不知道改了什么”的混乱状态

---

## 0. 总体原则

1. **default branch 永远保持可用**：`main` 或 `master` 只能放已验证、可编译、可说明的内容
2. **所有开发走分支**：禁止多人直接在 default branch 上开发
3. **一个任务一个 issue，一个分支解决一个主题**：不要一个分支同时修 CAN、改 README、重构 PID、调 UI
4. **小步提交，小步合并**：每个 PR 应该尽量小、职责单一、可 review
5. **代码与文档同步**：改接口必须改 README/注释/示例
6. **硬件测试必须记录**：涉及真实电机、舵机、底盘、机械臂的改动，必须写测试记录
7. **SDK 版本必须可追踪**：通过 tag、release、submodule commit 固定依赖版本

---

## 1. Git 与 GitHub 的区别

- **Git**：本地版本控制工具，负责提交、分支、合并、回退
- **GitHub**：远程代码托管与协作平台，负责 issue、PR、review、release、wiki/pages、actions

**基本关系**：

```text
本地工作区 --git commit--> 本地仓库 --git push--> GitHub 远程仓库
GitHub 远程仓库 --git pull/fetch--> 本地仓库/工作区
```

## 2. 分支模型

### 2.1 核心分支

```text
main       # 稳定分支，可编译、可教学、可作为对外展示版本
dev        # 可选：集成分支，用于多个 feature 合并前集中测试
```

只有两三个人时可以只保留 `main`，所有功能分支直接 PR 到 `main`；如果项目复杂、多人并行开发硬件和文档，可以增加 `dev`

### 2.2 功能分支命名

**推荐格式**：

```text
<type>/<scope>-<short-desc>
```

**常用 type**：

```text
feat      # 新功能
fix       # bug 修复
refactor  # 重构
docs      # 文档
test      # 测试
chore     # 工程杂项
sdk       # SDK 调整
chip      # 芯片适配
```

**示例**：

```text
feat/chassis-controller
fix/fdcan-rx-callback
refactor/motor-portops
docs/development-flow
sdk/infra-hfsm
chip/stm32-fdcan-notes
```

### 2.3 分支使用规则

1. 每个 issue 建一个分支
2. 分支从最新 `main` 或 `dev` 拉出
3. 开发前先同步远程：

```bash
git fetch origin
git switch main
git pull --ff-only origin main
```

4. 新建分支：

```bash
git switch -c feat/chassis-controller
```

5. 推送分支：

```bash
git push -u origin feat/chassis-controller
```

6. 分支完成后开 PR
7. PR 合并后删除远程分支和本地分支

## 3. 提交规范

### 3.1 Commit message 格式

**推荐使用**：

```text
<type>(<scope>): <summary>
```

**示例**：

```text
feat(chassis): add steer wheel inverse kinematics
fix(fdcan): avoid rx callback overwrite
refactor(motor): introduce PortOps interface
docs(flow): add development sequence checklist
test(parser): add frame parser boundary cases
```

### 3.2 type 说明

| type | 用途 |
|---|---|
| feat | 新功能 |
| fix | bug 修复 |
| refactor | 不改变行为的结构重构 |
| docs | 文档修改 |
| test | 测试相关 |
| chore | 构建、格式、依赖、杂项 |
| style | 格式调整，不改变逻辑 |
| perf | 性能优化 |
| revert | 回退提交 |

### 3.3 提交粒度

**推荐**：

- 一个 commit 只做一类事情；
- 能说明“为什么改”；
- 不提交编译产物；
- 不提交临时日志、大量无关格式化；
- 不把多个无关功能塞进一个 commit

## 4. 拉取、同步与冲突处理

### 4.1 日常同步

**推荐**：

```bash
git fetch origin
git status
git pull --ff-only origin main
```

**如果正在功能分支开发**：

```bash
git fetch origin
git switch feat/xxx
git rebase origin/main
```

**如果不熟悉 rebase 时使用 merge**：

```bash
git merge origin/main
```

**统一建议**：

- 优先使用 merge，减少误操作；
- 熟悉 Git 后可以使用 rebase 保持线性历史；
- 公共分支不要随便 rebase；
- 已 push 给别人共同使用的分支不要随便 force push

### 4.2 冲突处理原则

1. 先看冲突文件属于谁负责
2. 不理解对方代码时不要盲目覆盖
3. 解决冲突后必须重新编译或跑测试
4. 冲突解决应单独提交，message 写清楚

**示例**：

```bash
git status
# 手动编辑冲突文件
git add <file>
git commit -m "fix(merge): resolve chassis controller conflict"
```

## 5. Pull Request 标准

### 5.1 什么时候开 PR

**以下情况必须开 PR**：

- 合并到 `main`；
- 修改 SDK public API；
- 修改安全相关逻辑；
- 修改设备驱动；
- 修改芯片底层适配；
- 修改文档标准；
- 更新 submodule 指针

### 5.2 PR 标题格式

```text
[type][scope] summary
```

**示例**：

```text
[Refactor][motor] Introduce CAN PortOps interface
[Docs][flow] Add embedded development sequence
[Fix][fdcan] Fix RX callback dispatch
```

### 5.3 PR 描述模板

```markdown
## 变更内容

- 

## 影响范围

- [ ] app
- [ ] service
- [ ] device
- [ ] domain
- [ ] infra
- [ ] platform
- [ ] docs
- [ ] examples
- [ ] submodule

## 为什么这样改

-

## 测试记录

- 编译：通过 / 未通过
- PC 测试：
- 板级测试：
- 真实硬件测试：
- 测试 commit：

## 安全影响

- 是否影响电机/舵机/底盘/机械臂输出：是 / 否
- 是否影响 stop/brake/fault：是 / 否
- 是否需要低速验证：是 / 否

## 关联 Issue

Closes #
```

### 5.4 PR 合并要求

**合并前必须满足**：

- [ ] 至少 1 人 review；
- [ ] 没有明显跨层依赖；
- [ ] 代码能编译；
- [ ] 文档已同步；
- [ ] 涉及真实硬件的测试已记录；
- [ ] default branch 没有被破坏；
- [ ] submodule 指针变更可解释

## 7. Git submodule 使用标准

### 7.1 当前 submodule 策略

**当前AgroTech 协会标准采用以下策略**：

```text
成员项目通过 submodule 引入 Embedded-Electronic-Control-Standard，内部包含通用 sdk
具体芯片平台适配由成员项目按需额外添加 chip SDK submodule
```

**推荐项目结构**：

```text
My-Embedded-Project/
├── external/
│   ├── Embedded-Electronic-Control-Standard/
│   └── Embedded-Chip-STM32-HAL-SDK/
├── src/
└── README.md
```

### 7.2 在项目中添加标准仓库

```bash
git submodule add https://github.com/<user-or-org>/Embedded-Electronic-Control-Standard.git external/Embedded-Electronic-Control-Standard
git add .gitmodules external/Embedded-Electronic-Control-Standard
git commit -m "chore(submodule): add embedded electronic control standard"
```

### 7.3 克隆带 submodule 的项目

```bash
git clone --recurse-submodules <project-url>
```

**如果已经普通 clone**：

```bash
git submodule update --init --recursive
```

### 7.4 普通成员更新项目

**普通成员一般只需要更新到父项目锁定的标准版本**：

```bash
git pull --recurse-submodules
git submodule update --init --recursive
```

**这表示**：

```text
父项目记录当前应该使用哪个标准仓库 commit；
普通成员更新后，submodule 会回到父项目记录的版本
```

### 7.5 项目维护者升级标准仓库版本

**当项目需要使用标准仓库最新内容时，由项目维护者执行**：

```bash
cd external/Embedded-Electronic-Control-Standard
git fetch origin
git switch main
git pull --ff-only origin main

cd ../..
git add external/Embedded-Electronic-Control-Standard
git commit -m "chore(submodule): update embedded electronic control standard"
git push
```

然后通过 PR 合并

### 7.6 添加 chip SDK submodule

**如果项目使用 STM32**：

```bash
git submodule add https://github.com/<user-or-org>/Embedded-Chip-STM32-HAL-SDK.git external/Embedded-Chip-STM32-HAL-SDK
git add .gitmodules external/Embedded-Chip-STM32-HAL-SDK
git commit -m "chore(submodule): add stm32 hal chip sdk"
```

**如果项目使用 Renesas RA**：

```bash
git submodule add https://github.com/<user-or-org>/Embedded-Chip-Renesas-FSP-SDK.git external/Embedded-Chip-Renesas-FSP-SDK
git add .gitmodules external/Embedded-Chip-Renesas-FSP-SDK
git commit -m "chore(submodule): add renesas fsp chip sdk"
```

### 7.7 不推荐普通成员使用的命令

```bash
git submodule update --remote --recursive
```

该命令会让 submodule 直接追远程分支最新提交，可能绕过父项目锁定的已验证版本。普通成员应使用父项目记录的 submodule commit；只有项目维护者在升级标准版本时才应主动更新 submodule 指针

### 7.8 submodule 常见坑

1. **clone 后 submodule 目录为空**：执行 `git submodule update --init --recursive`；
2. **只 pull 父项目，标准仓库没更新**：执行 `git pull --recurse-submodules` 后再执行 `git submodule update --init --recursive`；
3. **在 submodule 内改了代码但没提交到标准仓库**：别人无法获取你的改动；
4. **项目维护者更新了 submodule 但忘记提交父项目指针**：别人仍然停留在旧版本；
5. **普通成员直接追 latest main**：可能导致本地标准版本与项目验证版本不一致；
6. **跨仓库频繁同时修改**：说明接口边界不稳定，应先在 issue 中讨论

## 8. GitHub Project 使用标准

Project 不存放正文标准，正文应放在仓库 Markdown 中

**Project 用于**：

- 规划标准撰写进度；
- 管理 SDK 开发任务；
- 管理 issue 和 PR；
- 追踪里程碑；
- 分配成员任务；
- 展示路线图

**推荐字段**：

| 字段 | 类型 | 用途 |
|---|---|---|
| Status | Single select | Todo / In Progress / Review / Done / Blocked |
| Type | Single select | Docs / Infra SDK / Device SDK / Algorithm SDK / Chip SDK / Example / Test |
| Layer | Single select | app / service / device / domain / infra / platform |
| Priority | Single select | P0 / P1 / P2 / P3 |
| Difficulty | Number | 1~5 |
| Owner | Assignee | 负责人 |
| Start Date | Date | 开始时间 |
| Target Date | Date | 目标完成时间 |
| Version | Text | v0.1 / v0.2 / v1.0 |
| Evidence | Text | PR、测试记录、演示视频、示例工程链接 |

**推荐视图**：

1. Board：按 Status 分组，日常推进；
2. Table：查看所有任务和负责人；
3. Roadmap：看 v0.1/v0.2/v1.0 节点；
4. Layer：按 app/service/device/domain/infra/platform 分组；
5. SDK：只看 SDK 任务

## 9. Release 与版本管理

### 9.1 文档版本

```text
Draft v0.1  # 初版，可讨论
Draft v0.2  # 有示例工程，可试用
v1.0        # AgroTech 协会正式标准
```

### 9.2 SDK 版本

**推荐语义化版本**：

```text
v0.1.0  # 初始可用
v0.2.0  # 增加模块
v1.0.0  # API 稳定
v1.1.0  # 向后兼容新增功能
v2.0.0  # 破坏性 API 修改
```

### 9.3 Release checklist

- [ ] README 更新；
- [ ] plan.md 更新；
- [ ] examples 可运行；
- [ ] 关键模块有测试；
- [ ] 项目 submodule 指针已固定；
- [ ] 网站已同步；
- [ ] release notes 写明 breaking changes

## 10. 禁止行为

- 禁止直接向 `main` 强推；
- 禁止在未说明的情况下 force push 公共分支；
- 禁止提交 `build/`、`.elf`、`.bin`、`.map` 等大多数构建产物，除非明确需要归档；
- 禁止提交私钥、token、WiFi 密码、个人路径；
- 禁止在未测试情况下合并硬件输出相关代码；
- 禁止大范围无意义格式化导致 review 困难；
- 禁止 submodule 内有未 push commit 就提交父仓库指针；
- 禁止绕过 PR 直接改标准正文

## 11. 新成员上手流程

### 11.1 参与标准仓库开发

1. 阅读 `README.md`；
2. 阅读 `plan.md`；
3. 阅读 `通用开发流文档.md`；
4. 阅读 `团队协作开发文档.md`；
5. 从 issue 中选择任务；
6. 新建分支；
7. 修改文档或 SDK；
8. 本地检查；
9. 提交 PR

### 11.2 参与具体项目开发

1. clone 项目：

```bash
git clone --recurse-submodules <project-url>
```

2. 如果已经普通 clone：

```bash
git submodule update --init --recursive
```

3. 阅读项目 README；
4. 阅读 `external/Embedded-Electronic-Control-Standard/README.md`；
5. 确认项目使用的 chip SDK；
6. 找一个 `good first issue`；
7. 新建分支；
8. 完成修改；
9. 编译/测试；
10. 提交 PR
